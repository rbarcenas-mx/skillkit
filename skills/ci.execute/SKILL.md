# ci.execute — CI Plan Execution

> **Language note**: All user-facing text below is in English. The orchestrator MUST present all interactions to the user in their language, translating as needed. Generated content (plan logs, reports) must also be in the user's language.

## Purpose

Read a `ci/{id}_tasks.md` plan file generated by `ci.prepare`, execute each task in order respecting dependencies, with save points, live progress bar, resume-after-failure, and per-task rollback. The orchestrator (this agent) handles plan selection, checkpoint detection, and delegates all execution to `run.py` (100% local, no model calls).

## Architecture

```
Orchestrator (this agent)
  ├─ 1. Resolve WORKDIR
  ├─ 2. Resolve model + show TOKEN_BUDGET
  ├─ 3. Select task plan file (ci/{id}_tasks.md)
  ├─ 4. Detect checkpoint → resume if applicable
  ├─ 5. Copy run.py → execute tasks one at a time
  │     ├─ run.py iterates tasks, shows progress bar on stderr
  │     ├─ Each task: display command, confirm, execute, log result
  │     ├─ Checkpoint saved after each task
  │     └─ run.py returns final JSON summary to stdout
  ├─ 6. Present execution report
  └─ 7. Token report
```

## Step 3 — Select task plan file

### 3.1 — List available plans

```bash
ls ci/*_tasks.md 2>/dev/null | grep -v '_completed'
```

Only show plans not yet closed with `_completed` by `ci.ship`.

### 3.2 — Select plan

- If multiple plans exist, use `question` tool to let user pick.
- If only one plan exists, use it directly.
- If `$ARGUMENTS` is provided, filter by that ID.
- If no plans found, abort and suggest running `ci.prepare` first.

### 3.3 — Read and validate plan

Read the selected file. Verify it contains:
- Header with `run_id`, `desc`, `total_tasks`
- Task section in `## EX-NNN` format
- Execution Log section (empty or with prior entries)

### 3.4 — Show execution plan

```
Plan selected: ci/001_20260619_1934_tasks.md
Total tasks:   N
Already done:  M (from execution log)
Pending:       N-M
```

Present task list summary and ask user to confirm.

## Step 4 — Detect checkpoint and resume state

```bash
cat /tmp/opencode/ci_execute_progress.json 2>/dev/null || echo '{"found": false}'
```

If a checkpoint exists:

```
Checkpoint found: yes
Last task done:  EX-00N
Completed:       N / M
```

If no checkpoint but the Execution Log has entries, compute completed count from the log.

`run.py` handles this internally — it reads the execution log, detects already-completed tasks (marked with ✅), and skips them. The orchestrator just needs to show the state to the user before starting.

## Step 5 — Copy and execute run.py

### 5.1 — Copy

```bash
cp "$HOME/.claude/skills/ci.execute/run.py" /tmp/opencode/ci_execute_run.py
```

### 5.2 — Execute

```bash
CI_TASKS_FILE="<absolute path to ci/{id}_tasks.md>" \
WORKDIR="$WORKDIR" \
python3 /tmp/opencode/ci_execute_run.py
```

Use `timeout=900000` (15 min).

`run.py` (100% local, no model):
- Parses the task plan
- Validates `.gitignore` before executing
- Detects already-completed tasks from Execution Log → skips them (resume)
- Iterates each pending task, showing progress bar `[████░░] XX% (N/M)` on stderr
- Respects task dependencies
- Displays each command before executing, asks for user confirmation (stdin)
- Executes commands via `subprocess`
- Captures short commit hash after each `git commit`
- Saves checkpoint to `/tmp/opencode/ci_execute_progress.json` after each task
- Updates execution log in the plan file with timestamp, status, duration, and hash
- If a task fails: logs error, shows to user, allows retry
- If a task contains `git push`, requires explicit confirmation before executing
- Outputs all progress to stderr (progress bar, task status, errors)
- Returns final JSON summary to stdout: `status`, `total`, `completed`, `failed`, `skipped`, `branch`, `commits`, `tracked_files`

**The orchestrator must relay run.py real-time stderr output to the user** — this is where the progress bar, task status, and errors appear line by line.

## Checkpoint & resume

- **Progress file**: `/tmp/opencode/ci_execute_progress.json`
- **Per-task checkpoint**: Each completed task is immediately written to the Execution Log in the plan file with ✅ status.
- **Resume mechanism**: When `run.py` starts, it reads the Execution Log. Tasks already marked ✅ are skipped. Tasks without entries (or marked ❌) are executed.
- **If power loss / interruption**: Re-run `ci.execute` — `run.py` detects completed tasks and resumes from the first pending one.
- **Rollback**: If a task fails and has a `rollback` command defined (not `N/A`), `run.py` offers to run it.

## Notes

- **Orchestrator model**: resolved via `resolve_model("ci.execute")` according to TOKEN_BUDGET. Actual task execution is 100% local via `run.py` (no model calls for execution).
- **Timeout**: `900000` (15 min) when running the script.
- **High risk**: `ci.execute` runs git commands that modify history (`git commit`, `git push`). Human confirmation is mandatory for `git push`.
- **Language**: All generated content must be in the user's language without accents or special characters.
